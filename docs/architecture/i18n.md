# Internationalization (i18n) Architecture

Kausal Paths uses three distinct translation mechanisms, each with its own
storage model and language code conventions. This document maps out where
each mechanism applies, how they interact, and how language codes are
converted between them.

## The Three Mechanisms

### 1. django-modeltrans (Django model fields)

Stores translations in a single JSON column called `i18n` on each model.
The "main" language value lives in the regular Django field; other languages
are stored as `{field}_{lang}` keys inside the JSON.

**Example storage** for a model with `name = "Hiilineutraali Tampere"`:
```json
{"name_en": "Carbon-neutral Tampere"}
```
The Finnish value is in the `name` column itself; English is in `i18n`.

**Models using `TranslationField`:**

| Model | Module | Translated fields | `default_language_field` |
|-------|--------|--------------------|--------------------------|
| `InstanceConfig` | `nodes/models.py` | `name`, `lead_title`, `lead_paragraph` | (global `LANGUAGE_CODE`) |
| `NodeConfig` | `nodes/models.py` | `name`, `short_description`, `description`, `goal` | `instance__primary_language` |
| `BaseOrganization` | `kausal_common/organizations/models.py` | `name`, `abbreviation` | `primary_language_lowercase` |
| `BaseOrganizationClass` | `kausal_common/organizations/models.py` | `name` | (global) |
| `Dimension` | `kausal_common/datasets/models.py` | `name` | (global) |
| `DimensionCategory` | `kausal_common/datasets/models.py` | `label` | (global) |
| `DatasetSchema` | `kausal_common/datasets/models.py` | `name` | (global) |
| `DatasetMetric` | `kausal_common/datasets/models.py` | `label`, `unit` | (global) |
| `BasePerson` | `kausal_common/people/models.py` | `title` | `organization__primary_language_lowercase` |

**Language code format:** `en`, `fi_fi`, `de_ch` (underscore separator,
all-lowercase region). This is the "modeltrans" format.

**Settings** (`paths/settings.py`):
```python
MODELTRANS_AVAILABLE_LANGUAGES = [x[0].lower() for x in LANGUAGES]
MODELTRANS_FALLBACK = {
    'default': (),
    'de-ch': ('de',),
}
```

### 2. Wagtail Locales (`wagtail-localize`)

Wagtail's own translation system, based on `Locale` objects in the database.
Used for CMS Page trees. In this codebase, `PathsPage` (in `pages/models.py`)
does **not** use Wagtail's `TranslatableMixin`; instead it manually manages
an `i18n` JSONField alongside modeltrans. The Wagtail locale machinery is
enabled (`WAGTAIL_I18N_ENABLED = True`) and `wagtail_localize` is installed,
but its primary role is admin UI language switching rather than model-level
translation storage.

**Language code format:** `en`, `fi-FI`, `es-US` (hyphen, uppercase region).

### 3. `TranslatedString` (Pydantic models / computation layer)

A custom type in `kausal_common/i18n/pydantic.py` that holds a
`dict[str, str]` mapping language codes to translations. Used in all Pydantic
`BaseModel` subclasses that need translated text: node specs, action configs,
scenario definitions, and anything stored in `SchemaField` JSON columns.

**Language code format (internal):** ISO / RFC 5646 — `en`, `fi-FI`, `es-US`
(hyphen, uppercase region).

**Key classes:**
- `TranslatedString` — the value type itself
- `I18nBaseModel` — Pydantic `BaseModel` subclass with `arbitrary_types_allowed`
- `I18nContext` — a `ContextVar`-based context that specifies the default and
  other languages for the current operation

**Context requirement:** A `TranslatedString` generally requires an active
`I18nContext` when created from a plain `str`, so the system knows which
language the string is in. There are three ways to satisfy this:

1. **Explicit context:** `with set_i18n_context('fi', ['en']): ...`
2. **Lazy Django strings:** `TranslatedString(_('some text'))` defers
   evaluation until an `I18nContext` is available.
3. **Dict initialization:** `TranslatedString(en='Hello', fi='Moi')` —
   language codes are explicit, so no context needed.

**Serialization invariant:** Pydantic models always serialize
`TranslatedString` values as explicit `{language: value}` dicts, even when
there is only one language. Plain strings are still accepted as an input
convenience when an `I18nContext` is active, but persisted JSON should not
depend on a default-language context to be interpreted.

**Interaction with modeltrans:**
- `get_modeltrans_attrs_from_str()` converts a `TranslatedString` into
  modeltrans-shaped keys for writing to the DB.
- `get_translated_string_from_modeltrans()` reads a modeltrans `i18n` JSON
  field and reconstructs a `TranslatedString`.

**`InstanceSpec` self-contextualization:** `InstanceSpec` (the top-level
Pydantic model stored on `InstanceConfig.spec`) carries `primary_language`
and `other_languages` fields. A `model_validator(mode='wrap')` reads these
fields and sets an `I18nContext` for the duration of validation, so all
nested `TranslatedString` fields are validated with the correct language
context — including when Pydantic deserializes the JSON from the database.


## Language Code Formats

Five different notations exist because each library chose its own convention.
All conversions go through `convert_language_code()` in
`kausal_common/i18n/helpers.py`.

| Format name | Separator | Region case | Examples | Used by |
|-------------|-----------|-------------|----------|---------|
| `'iso'` / `'kausal'` | `-` | UPPER | `en`, `fi-FI`, `de-CH` | `TranslatedString` internals, YAML configs |
| `'django'` | `-` | lower | `en`, `fi-fi`, `de-ch` | Django `LANGUAGES`, `get_language()` |
| `'modeltrans'` | `_` | lower | `en`, `fi_fi`, `de_ch` | modeltrans `i18n` JSON keys |
| `'wagtail'` | `-` | UPPER | `en`, `fi-FI`, `de-CH` | Wagtail `Locale` objects |
| `'next.js'` | `-` | UPPER | `en`, `fi-FI`, `de-CH` | Frontend (i18next) |

Note that `iso`, `wagtail`, and `next.js` are identical in practice. The
distinction exists so code is self-documenting about *why* a particular
format is needed.

The conversion function:
```python
from kausal_common.i18n.helpers import convert_language_code

convert_language_code('de-ch', 'iso')        # → 'de-CH'
convert_language_code('de-CH', 'modeltrans')  # → 'de_ch'
convert_language_code('de_ch', 'django')      # → 'de-ch'
```

Base-only codes (e.g. `en`, `fi`) are unchanged across all formats.


## Global Language Configuration

Defined in `paths/settings.py`:

```python
LANGUAGES = (
    ('en', _('English')),
    ('fi', _('Finnish')),
    ('sv', _('Swedish')),
    ('de', _('German')),
    ('de-CH', _('German (Switzerland)')),
    ('cs', _('Czech')),
    ('da', _('Danish')),
    ('pl', _('Polish')),
    ('lv', _('Latvian')),
    ('es-US', _('Spanish (United States)')),
    ('el', _('Greek')),
)
LANGUAGE_CODE = 'en'  # global default
```

Each `InstanceConfig` then specifies its own `primary_language` and
`other_languages` (a `ChoiceArrayField`) from this global set.


## YAML Instance Configs

Instance YAML files use field-name suffixes for translations:

```yaml
id: tampere
default_language: fi
supported_languages: [en]
name: Hiilineutraali Tampere 2030
name_en: Carbon-neutral Tampere 2030
```

The suffix language codes follow Django format (lowercase). The
`default_language` and `supported_languages` values also use Django format.


## Admin UI

`kausal_common/i18n/forms.py` provides `LanguageAwareAdminModelForm`, which
inspects the instance's language configuration and prunes form fields for
irrelevant languages. Panel classes in `kausal_common/i18n/panels.py`
(`PrimaryLanguagePanel`, `TranslatedLanguagePanel`, `TranslatedFieldPanel`)
render language-specific fields with appropriate headings.


## Common Pitfalls

1. **Missing I18nContext at import time.** Module-level code that constructs
   `TranslatedString` from a plain `str` will fail if no context is set.
   Use Django's lazy strings (`_('...')`) for class-level attributes like
   `explanation` or parameter `label` fields.

2. **Language code mismatch.** Passing a Django-format code (`de-ch`) where
   modeltrans format (`de_ch`) is expected (or vice versa) silently creates
   orphan translation keys. Always use `convert_language_code()`.

3. **Fallback gaps.** `MODELTRANS_FALLBACK` only defines `de-ch → de`.
   Other regional variants have no fallback chain. `TranslatedString` has
   its own fallback logic (strip region, try base language) which may
   diverge from modeltrans behavior.
